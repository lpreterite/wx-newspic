# 阶段门控清单

> Phase Gate Checklists

---

## 1. 概述

本文档提供 AI 软件研发流程各阶段 Gate 的验收检查清单，用于确保每个关键节点的交付物符合质量标准。

### 1.1 Gate 总览

| Gate | 名称 | 验收内容 | 通过条件 |
|------|------|----------|----------|
| **Gate 1** | PRD 验收 | 用户故事、线路图、验收标准、原型图 | 人类确认 |
| **Gate 2** | 里程碑验收 | 技术方案、用户方案、里程碑划分 | 人类确认 |
| **Gate 3** | 可使用软件验收 | 单元测试、E2E测试、人工验收 | 人类确认 |
| **Gate 4** | 可维护软件验收 | 已部署软件、使用说明文档 | 人类最终确认 |

---

## 2. Gate 1：PRD 验收

**验收内容**：PRD 中的用户故事、用户使用线路图、验收标准、原型图

**检查清单**：

### G1-01 用户故事

**前置依赖**：线路图设计（参见 [02-process.md 附录](./02-process.md#附录流程与产出物对照表)）

```
□ 1. 用户故事覆盖所有核心功能
□ 2. 用户故事描述清晰、可理解
□ 3. 用户故事从用户视角描述
□ 4. 用户故事包含验收标准
□ 5. 优先级划分合理（Must/Should/Could/Won't）
□ 6. 用户线路图包含主要使用路径
□ 7. 核心场景在线路图中覆盖
□ 8. 路径之间的跳转关系清晰
```

### G1-02 验收标准

**前置依赖**：G1-01 用户故事

```
□ 1. 验收标准与用户故事对应
□ 2. 每个用户故事都有验收标准
□ 3. 验收标准可测试
□ 4. 验收标准无歧义
□ 5. 验收标准包含正向和负向场景
□ 6. 边界条件有明确说明
□ 7. 性能要求有量化指标
```

### G1-03 原型图

**前置依赖**：G1-01 用户故事、G1-02 验收标准

```
□ 1. 原型图覆盖所有核心页面
□ 2. 原型图与用户故事对应
□ 3. 交互流程与线路图一致
□ 4. 页面元素和布局合理
□ 5. 关键状态（空态、加载态、错误态）有体现
```

### G1-04 跨章节一致性检查

**前置依赖**：G1-01 用户故事、G1-02 验收标准、G1-03 原型图

```
□ 1. Non-Goals 中的「不做」列表与验收标准无冲突
□ 2. 关键术语在全文定义一致（尤其是 Goals、Non-Goals、验收标准之间的术语）
□ 3. 用户故事中的表述与验收标准中的术语一致
□ 4. 原型图与文字描述无矛盾
```

### G1-05 测试策略

**前置依赖**：G1-03 原型图

```
□ 1. 测试策略已明确（单元测试/集成测试/E2E 测试划分方案）
□ 2. 各测试类型的责任人和归属范围已定义
□ 3. 测试环境要求已识别
```

**关键审查点**：
- 用户故事是否完整、清晰
- 线路图是否覆盖核心场景
- 标准可测试、无歧义
- 原型图是否符合需求

**验收方式**：逐步审查核对

## 3. Gate 2：里程碑验收

**验收内容**：技术方案、用户方案、里程碑划分

**检查清单**：

### G2-01 技术方案

**前置依赖**：Gate 1 验收通过

```
□ 1. 系统架构设计清晰
□ 2. 模块划分合理、职责明确
□ 3. 技术选型有充分依据
□ 4. 技术选型符合项目约束（时间、成本、技术栈）
□ 5. 接口定义完整、无歧义
□ 6. 数据模型设计合理
□ 7. 错误处理机制明确
□ 8. 日志和监控方案到位
□ 9. 考虑了扩展性和维护性
□ 10. 考虑了安全性和性能
□ 11. 依赖项管理清晰
□ 12. 部署方案可行
```

### G2-02 用户方案

**前置依赖**：Gate 1 验收通过

```
□ 1. 设计稿覆盖所有页面
□ 2. 设计稿覆盖所有状态（默认、悬停、点击、禁用等）
□ 3. 标注规范完整
□ 4. 间距、颜色、字体有明确说明
□ 5. 尺寸规格覆盖响应式断点
□ 6. 交互说明清晰（组件状态、过渡动画、微交互）
□ 7. 图标资源齐全
□ 8. 设计标注已导出（Zeplin/Figma）
□ 9. 特殊效果有说明
□ 10. 符合设计系统规范
□ 11. 一致性检查通过
□ 12. 可访问性考虑
```

### G2-03 里程碑划分

**前置依赖**：G2-01 技术方案、G2-02 用户方案

```
□ 1. 阶段划分合理、可执行
□ 2. 各阶段交付物清单完整
□ 3. 里程碑验收标准清晰
□ 4. 时间估算合理
□ 5. 依赖关系已识别
```

---

**关键审查点**：
- 技术方案是否合理、可实现
- 用户方案是否符合体验要求
- 里程碑划分是否合理

**验收方式**：逐步审查核对

---

## 4. Gate 3：可使用软件验收

**验收内容**：代码实现、测试通过、人工验收

**检查清单**：

### G3-01 代码实现完成

**前置依赖**：任务编排（参见执行阶段检查清单 §6）

```
□ 1. 所有用户故事已实现
□ 2. 技术方案全部实现
□ 3. 设计稿全部实现
□ 4. 验收标准全部通过
```

### G3-02 单元/冒烟测试通过

**前置依赖**：G3-01 代码实现完成

```
□ 1. 单元测试通过
□ 2. 集成测试通过
□ 3. 功能测试通过
```

### G3-03 集成/E2E 测试通过

**前置依赖**：G3-02 单元/冒烟测试通过

```
□ 1. E2E 测试通过
□ 2. 回归测试通过
□ 3. 性能测试通过（如适用）
□ 4. 安全测试通过（如适用）
```

### G3-04 人工验收通过

**前置依赖**：G3-03 集成/E2E 测试通过

```
□ 1. 人工验收通过
```

### G3-05 CCPM 阻塞消除

**前置依赖**：G3-01~G3-04 全部通过

```
□ 1. CCPM 关键链上的阻塞已全部消除
```

**关键审查点**：
- 软件功能是否完整实现
- 测试是否全面覆盖
- 人工验收是否确认

**验收方式**：自动化测试 + 人类确认

## 5. Gate 4：可维护软件验收

**验收内容**：已部署软件、使用说明文档

**检查清单**：

### G4-01 软件已部署

**前置依赖**：Gate 3 验收通过

```
□ 1. 软件已部署到目标环境
□ 2. 部署包就绪
□ 3. 上线验证通过
```

### G4-02 使用说明文档已完成

**前置依赖**：G4-01 软件已部署

```
□ 1. 用户文档齐全
□ 2. 运维文档齐全
```

### G4-03 运维/数据迁移/培训

**前置依赖**：G4-02 使用说明文档已完成

```
□ 1. 数据迁移方案就绪（如适用）
□ 2. 培训材料就绪（如适用）
```

**关键审查点**：
- 软件是否可交付
- 是否满足最初的需求目标
- 文档是否齐全

**验收方式**：人类最终确认

## 6. 执行阶段检查清单

### 5.1 任务启动检查

```
□ 1. 任务描述清晰
□ 2. 验收标准明确
□ 3. 依赖项已识别
□ 4. 资源已分配
□ 5. 截止时间已确定
□ 6. 风险已评估
□ 7. Issue 已关联看板和 Milestone
□ 8. 状态已在看板中更新为"进行中"
□ 9. 当前领域/项目的进行中/待验收 Issue 状态已查阅，确认无更优先事项
□ 10. 相关关键文档已查阅（架构/设计/PRD等），当前方案与文档记录一致
```

### 5.2 任务完成检查

#### 自动触发机制

当以 Issue 为载体的任务所有检查项（不含"变更已提交"）均通过时，Agent 按以下规则自动执行：

**触发条件**：

```
□ 1. 代码编写完成
□ 2. 代码审查通过
□ 3. 单元测试通过
□ 4. 集成测试通过（如适用）
□ 5. 功能测试通过
□ 6. 文档已更新
□ 7. 本次任务若产生影响关键文档的决策：已更新对应文档
```

决策结果更新映射：

| 决策类型 | 更新目标文档 |
|----------|-------------|
| 技术选型/架构变更 | `docs/engineering/` 或 `docs/architecture/ARCHITECTURE.md` |
| 设计风格/布局/信息架构 | `docs/design/` |
| 需求/体验/业务流程变更 | `docs/product/PRD.md` |

**自动执行动作**：

| 顺序 | 动作 | 说明 |
|------|------|------|
| 1 | 提交变更 | `git add` + `git commit` + `git push`，commit message 引用 Issue 编号（§9.3） |
| 2 | 更新 STATUS.md | 标记 Issue 为完成，追加「最近更新」条目（格式见 `05-deliverables.md §5.1`） |
| 3 | 更新看板状态 | 将 Issue 移入「待验收」列 |

**确认模式演进**（状态记录在项目 STATUS.md 的「自动化设置」小节）：

| 阶段 | 条件 | Agent 行为 |
|------|------|------------|
| 冷启动 | 自动化设置为 `disabled` 或无记录 | 执行检查后询问用户"是否提交？"，确认后执行 |
| 学习期 | 累计人工确认 ≥ 2 次 | 询问用户"是否启用自动提交？" |
| 自动模式 | 自动化设置为 `enabled` | 满足条件后静默执行，无须询问 |

> 用户可随时通过 Issue 要求关闭自动提交。

### 5.3 测试执行检查

```
□ 1. 测试计划就绪
□ 2. 测试用例覆盖验收标准
□ 3. 测试环境就绪
□ 4. 测试执行记录完整
□ 5. Bug 已记录和跟踪
□ 6. 回归测试计划就绪
```

### 5.4 设计交接检查清单

| 检查项 | 说明 | 负责方 |
|--------|------|--------|
| 1. 设计稿完整 | 所有页面和状态均已覆盖 | UI/UX Agent |
| 2. 标注清晰 | 开发人员可无歧义地实现 | UI/UX Agent |
| 3. 标注已导出 | Zeplin/Figma 标注已同步 | UI/UX Agent |
| 4. 特殊说明 | 动画、过渡等特殊效果已说明 | UI/UX Agent |
| 5. 资源齐全 | 图标、字体等资源已提供 | UI/UX Agent |
| 6. 设计已定稿 | 无待解决的疑问或争议 | Orchestrator Agent |

### 5.5 Issue 与看板就绪检查

```
□ 1. 里程碑已拆分为可执行的 Issues
□ 2. 每个 Issue 包含明确的执行计划
□ 3. 每个 Issue 有可测试的验收标准
□ 4. Issue 已关联 Milestone 和对应里程碑
□ 5. Issue 优先级已标注（P0/P1/P2）
□ 6. 超过 3 天工作量的 Issue 已拆分为 Sub-issue
□ 7. 看板已建立，状态列完整（待办/进行中/待验收/已完成）
□ 8. 看板 WIP 限制已设置（每 Agent ≤ 3 个）
□ 9. Issue 已同步到看板的对应状态列
```

### 5.6 测试与依赖就绪检查

```
□ 1. 每个功能 Issue 包含单元测试 AC（覆盖度要求 ≥ 80%）
□ 2. 集成测试 Issue 已创建（`{Phase}-T`）并关联 Milestone
□ 3. E2E 测试 Issue 已创建（`{Phase}-T`）并关联 Milestone
□ 4. 依赖关系已按 CCPM 标注（dependsOn / blockedBy / criticalChain）
□ 5. 关键链（criticalChain = true）上的阻塞已识别
```
