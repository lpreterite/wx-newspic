## 1. 概述

### 1.1 软件研发流程阶段

AI 软件研发流程分为两大阶段：

| 阶段 | 核心目标 | 人类参与度 |
|------|----------|-----------|
| **打磨阶段** | 定义"做正确的事" | 深度参与 |
| **执行阶段** | "正确地做事" | 关键节点介入 |

## 2. 打磨阶段详解

### 2.1 打磨阶段角色分工

| 阶段 | 人类负责 | Agent 负责 |
|------|----------|------------|
| 需求收集 | ✅ 用户诉求原始语料收集 | - |
| 需求分析 | - | PO Agent：动机提取、线路图设计 |
| PRD 起草 | - | PO Agent：用户故事、验收标准、原型图 |
| Roadmap | 验收 | PO Agent + PM Agent：产品路线图 |
| 技术方案 | 验收 | PM Agent + Developer Agent：起草 |
| 用户方案 | 验收 | UI/UX Agent：设计规范+设计稿 |
| 里程碑规划 | 验收 | PO Agent + PM Agent：Milestone 划分 |

### 2.2 打磨阶段 Gate

#### Gate 1：PRD 验收

**验收内容**：
- 用户故事
- 用户使用线路图
- 验收标准
- 原型图

**关键审查点**：
- 用户故事是否完整、清晰
- 线路图是否覆盖核心场景
- 验收标准是否可测试
- 原型图是否符合需求

**验收方式**：逐步审查核对

| 步骤 | Agent 行为 | 人类行为 |
|------|-----------|----------|
| 1 | 提取用户故事清单，展示核心场景 | 确认场景覆盖度 |
| 2 | 提取验收标准，标注可测试性 | 判断标准是否清晰、可验证 |
| 3 | 呈现用户线路图 | 确认路径是否完整、合理 |
| 4 | 记录人类反馈，生成修订建议 | 提出修改意见 |
| 5 | 迭代更新文档 | 复审确认通过 |

#### Gate 2：里程碑验收

**验收内容**：
- 技术方案（系统设计、技术选型）
- 用户方案（设计规范、设计稿）
- 里程碑划分

**关键审查点**：
- 技术方案是否合理、可实现
- 用户方案是否符合体验要求
- 里程碑划分是否可执行

**验收方式**：逐步审查核对

| 步骤 | Agent 行为 | 人类行为 |
|------|-----------|----------|
| 1 | 提取技术方案关键设计点 | 判断方案是否合理 |
| 2 | 提取用户方案核心设计决策 | 判断是否符合体验要求 |
| 3 | 提取里程碑划分方案 | 判断里程碑是否可执行 |
| 4 | 记录人类反馈，生成修订建议 | 提出修改意见 |
| 5 | 迭代更新文档 | 复审确认通过 |

### 2.3 执行阶段 Gate 3：可使用软件验收

**验收内容**：
- 通过单元测试
- 通过端到端测试（E2E）
- 人工验收完成

**关键审查点**：
- 单元测试覆盖率是否达标
- E2E 测试是否覆盖核心用户流程
- 软件功能是否符合验收标准

**验收方式**：自动化测试 + 人类确认

### 2.4 重做机制

打磨阶段任一 Gate 验收不通过时：
- **重做成本由 Agent 承担**
- Agent 重新走对应流程
- 人类在新一轮迭代中逐步验收

## 3. 执行阶段详解

### 3.1 执行阶段特点

- **Agent 自主协作**：Developer Agent 与 Tester Agent 在 PM Agent 协调下自主执行
- **人类关键介入**：在 Gate 3（可使用软件验收）和 Gate 4（可维护软件验收）进行确认
- **PM Agent 协调**：处理分歧、监控进度、触发升级

### 3.2 执行阶段流程

```
需求就绪
    │
    ▼
PM Agent 分配任务
    │
    ▼
┌─────────────────┐     ┌─────────────────┐
│ Developer Agent │◄───►│  Tester Agent   │
│   执行开发       │     │   执行测试       │
└────────┬────────┘     └────────┬────────┘
         │                      │
         │    PM Agent 协调     │
         │    （基于风险优先级） │
         │                      │
         └──────────┬───────────┘
                    │
                    ▼
            分歧解决机制
                    │
                    ▼
         ┌─────────────────┐
         │   验收确认       │  ← PO 确认
         └────────┬────────┘
          │
                   ▼
           【Gate 3】可使用软件验收
           自动化测试 + 人类确认
                    │
                    ▼
           ┌─────────────────┐
           │   上线发布       │
           └────────┬────────┘
                    │
                    ▼
           【Gate 4】可维护软件验收
           已部署 + 文档完成
```

### 3.3 分歧处理机制

当 Developer Agent 与 Tester Agent 出现分歧时：
- 由 PM Agent 根据风险优先级处理
- 低风险：Agent 自行协调
- 中风险：PM Agent 介入协调
- 高风险：升级人类 PM 决策

### 3.4 交付阶段 Gate 4：可维护软件验收

**验收内容**：
- 已部署给用户使用的软件
- 使用说明文档

**关键审查点**：
- 软件是否已部署并可正常访问
- 用户文档是否完整、可理解

**验收方式**：人类最终确认

## 4. 关键阈值

| 指标 | 警告阈值 | 升级阈值 | 处理策略 |
|------|----------|----------|----------|
| 任务延期 | > 1 天 | > 3 天 | 提醒 → 协调 → 升级 |
| 阻塞项时长 | > 4 小时 | > 24 小时 | 提醒 → 升级 |
| 测试失败率 | > 10% | > 30% | 提醒 → 暂停合并 |
| 风险概率变化 | +10% | +30% | 重新评估优先级 |
| 里程碑偏差 | < 5% | < 15% | 密切监控 → 启动补救 |
