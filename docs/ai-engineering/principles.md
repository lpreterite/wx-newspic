## 1. 概述

### 1.1 什么是 AI 软件研发

AI 软件研发是指在 AI 原生开发模式下，人与 Agent 形成混合智能体，共同完成软件产品的需求到交付全流程。

**核心特征**：
- 人类专注于需要判断力、创造力和责任承担的决策
- Agent 专注于执行、信息聚合和流程自动化
- 阶段门控作为质量关卡，确保关键节点的质量

### 1.2 核心定位

```
┌─────────────────────────────────────────────────────────────┐
│                     人机协作核心模式                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐ │
│   │   信息聚合    │    │   流程驱动    │    │   决策支持    │ │
│   │   (Gather)   │    │ (Orchestrate) │    │   (Advise)   │ │
│   └──────────────┘    └──────────────┘    └──────────────┘ │
│                                                              │
│   人类角色：判断力 + 创造力 + 最终责任                         │
│   Agent 角色：执行 + 信息聚合 + 流程自动化                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 人机分工原则

| 职责 | Agent 承担 | 人类承担 |
|------|-----------|----------|
| 进度收集 | ✅ 自动汇总各渠道状态 | 解读复杂情境、做判断 |
| 风险识别 | ✅ 基于规则/阈值预警 | 评估影响、做应对决策 |
| 任务分配 | ✅ 基于技能/负载推荐 | 最终分配、与成员沟通 |
| 阻塞解决 | ✅ 升级提醒、协助追踪 | 亲自协调、推动解决 |
| 文档更新 | ✅ 按规则同步文档 | 审批重大变更 |
| 进度汇报 | ✅ 生成标准化报告 | 撰写解读性结论 |
| 需求优先级 | 提供数据支持 | 最终决策 |
| 团队协调 | 安排会议、发送提醒 | 主持评审、推动共识 |

## 2. 阶段门控质量控制机制

### 2.1 什么是阶段门控

阶段门控（Stage Gate）是里程碑转换点的验收检查，是质量控制的核心机制。

```
┌──────────────────────────────────────────────────────────────────────┐
│                          阶段门控示意图                                │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ════════════════════════ 打磨阶段 ═════════════════════════════════  │
│  最终交付物：PRD + 里程碑                                              │
│                                                                      │
│  PRD（用户使用线路图、用户故事、用户故事地图、原型图、验收标准）         │
│  Roadmap → 技术方案 → 设计规范+设计稿 → Milestone                      │
│                                                                      │
│  ┌─────────┐                         ┌─────────┐                     │
│  │ Gate 1  │ ─────────────────────▶ │ Gate 2  │                     │
│  │ PRD验收  │    技术方案/用户方案     │里程碑验收│                     │
│  └─────────┘    /里程碑起草           └─────────┘                     │
│       │                                   │                           │
│       │ 通过                              │ 通过                      │
│       ▼                                   ▼                           │
│  ════════════════════════ 执行阶段 ════════╪═════════════════════════  │
│                                           │                           │
│  ┌─────────┐                              │                           │
│  │ Gate 3  │ ◀────────────────────────────┘                           │
│  │可使用软件│  Agent 自主协作执行                                      │
│  │  验收    │                                                          │
│  └─────────┘                                                          │
│       │                                                               │
│  ═══════╪════════════════════ 交付阶段 ═══════════════════════════════  │
│       │                                                               │
│  ┌─────────┐                                                          │
│  │ Gate 4  │                                                          │
│  │可维护软件│  人类最终确认                                            │
│  │  验收    │                                                          │
│  └─────────┘                                                          │
│                                                                      │
│  人类深度参与（打磨） ──▶ Agent 自主协作（执行） ──▶ 人类最终确认（交付）│
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 打磨阶段交付物

打磨阶段也称为"需求打磨"，目标定义"做正确的事"。打磨阶段的最终交付物是 **PRD + 里程碑**。

打磨阶段包含以下交付物：

| 交付物 | 说明 |
|--------|------|
| **PRD** | 用户使用线路图、用户故事、用户故事地图、原型图、验收标准等内容 |
| **Roadmap** | 产品路线图 |
| **技术方案** | 系统设计、技术选型 |
| **设计规范+设计稿** | UI/UX 设计 |
| **里程碑** | 阶段划分与验收标准 |

### 2.3 门控的作用

| 作用 | 说明 |
|------|------|
| **质量关卡** | 在关键节点确保产出物符合标准 |
| **决策节点** | 为人类提供介入时机，确保正确方向 |
| **风险控制** | 防止低质量产出进入下一阶段 |
| **迭代节奏** | 通过验收不通过触发重做机制 |

### 2.4 执行阶段验收机制

执行阶段的目标是"正确地做事"。每周开发一个或多个功能后，通过单元测试或端到端测试进行验收：

- **单元测试**：基于技术方案设计，验证各模块的正确性
- **端到端测试**：基于用户故事和验收标准设计，验证完整功能流程

### 2.5 打磨阶段门控

打磨阶段有两个 Gate，分别对应两个关键交付物的确认：

| Gate | 验收内容 | 关键审查点 | 通过条件 |
|------|----------|-----------|----------|
| **Gate 1** | PRD 文档（用户故事、线路图、验收标准、原型图） | 用户故事完整、线路图覆盖核心场景、标准可测试、原型图符合需求 | 人类确认 |
| **Gate 2** | 里程碑文档（技术方案、用户方案、里程碑划分） | 方案合理可实现、设计符合体验要求、里程碑可执行 | 人类确认 |

### 2.6 Gate 3：可使用软件验收

执行阶段的门控是 **可使用软件的验收**（Gate 3）。

可使用的软件包含：
- 通过单元测试
- 通过端到端测试
- 人工验收完成的软件

### 2.7 Gate 4：可维护软件验收

交付阶段的门控是 **可维护软件的验收**（Gate 4）。

可维护的软件包含：
- 已部署的可给用户使用的软件
- 包含使用说明文档

### 2.8 重做机制

打磨阶段任一 Gate 验收不通过时：

- **重做成本由 Agent 承担**
- Agent 重新走对应流程
- 人类在新一轮迭代中逐步验收

## 3. 逐步审查核对模式

### 3.1 核心原则

Agent 做"信息提取和呈现"，人类做"深度理解和判断"，通过多轮对话逐步达成验收共识。

### 3.2 PRD 验收流程

| 步骤 | Agent 行为 | 人类行为 |
|------|-----------|----------|
| 1 | 提取用户故事清单，展示核心场景 | 确认场景覆盖度 |
| 2 | 提取验收标准，标注可测试性 | 判断标准是否清晰、可验证 |
| 3 | 呈现用户线路图和原型图 | 确认路径是否完整、合理；原型图是否符合需求 |
| 4 | 记录人类反馈，生成修订建议 | 提出修改意见 |
| 5 | 迭代更新文档 | 复审确认通过 |

### 3.3 方案验收流程

| 步骤 | Agent 行为 | 人类行为 |
|------|-----------|----------|
| 1 | 提取技术方案关键设计点 | 判断方案是否合理 |
| 2 | 提取用户方案核心设计决策 | 判断是否符合体验要求 |
| 3 | 呈现方案间的依赖关系 | 确认接口契约 |
| 4 | 记录人类反馈，生成修订建议 | 提出修改意见 |
| 5 | 迭代更新文档 | 复审确认通过 |

## 4. Agent 能力边界

### 4.1 应做事项（Can Do）

| 类别 | 具体能力 |
|------|----------|
| **信息处理** | 聚合多源状态、检测变更、生成摘要、更新文档 |
| **流程执行** | 触发检查点、推进状态机、发送提醒、安排会议 |
| **数据分析** | 计算进度偏差、预测趋势、评估影响 |
| **协作支持** | 传递上下文、协调多 Agent、记录决策 |
| **规则遵循** | 按预设规则自动决策、更新文档、处理标准化流程 |

### 4.2 不做事项（Won't Do）

| 类别 | 具体限制 | 原因 |
|------|----------|------|
| **人员管理** | 不做绩效评估、不做人员调度决策 | 超出信息范围，需人工判断 |
| **财务决策** | 不审批预算、不做成本权衡 | 需业务上下文 |
| **客户沟通** | 不直接对接外部客户 | 授权和语境限制 |
| **创造性方案** | 不替代产品经理做功能设计 | 核心人机协作价值 |
| **情绪处理** | 不处理团队冲突、不做心理疏导 | 需要人情练达 |

### 4.3 能力演进路径

```
当前 MVP 阶段          下一阶段              未来理想
─────────────────     ──────────────────    ──────────────────
• 文档状态跟踪         • 主动风险预测         • 自主驱动迭代
• 任务进度监控         • 智能任务分配         • 自动化学位调整
• 简单升级提醒         • 多项目协调          • 跨项目资源优化
• 标准化报告生成       • 集成更多工具         • 预测性干预
```

## 5. 决策分级

### 5.1 授权范围（Agent 可自动决策）

| 类型 | 说明 |
|------|------|
| 文档同步更新 | 按规则自动更新相关文档 |
| 例行状态汇总 | 定时生成进度报告 |
| 会议提醒与议程安排 | 按时发送提醒 |
| 阻塞项升级提醒 | 超过阈值自动提醒 |
| CI/CD 失败通知 | 触发告警 |

### 5.2 建议范围（需人类决策）

| 类型 | 说明 |
|------|------|
| 里程碑范围调整 | 影响整体计划 |
| 风险应对策略 | 高风险处置 |
| 任务重新分配 | 人力调度 |
| 优先级变更 | 范围变化 |
| 介入协调冲突 | 需要人际沟通 |
