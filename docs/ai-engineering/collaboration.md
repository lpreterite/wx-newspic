# 人机协作协议

> Human-Agent Collaboration Protocol

**所属目录**：`ai-engineering/guide/`
**文档状态**：草稿
**当前版本**：v0.2
**发布日期**：2026-04-04

---

## 1. 概述

本文档定义 AI 软件研发流程中人机协作的具体协议，包括协作模式、逐步审查核对机制、升级机制和冲突处理。

### 1.1 协作核心原则

| 原则 | 说明 |
|------|------|
| **人类深度参与** | 人类专注于判断力、创造力、责任承担 |
| **Agent 执行增强** | Agent 专注于执行、信息聚合、流程自动化 |
| **阶段门控质量** | 关键节点设置验收检查，确保质量 |
| **渐进式验收** | 通过多轮对话逐步达成验收共识 |

### 1.2 角色定位

| 角色 | 定位 |
|------|------|
| **Product Owner (PO)** | 人类，需求负责、验收把关 |
| **Product Manager (PM)** | 人类，流程协调、进度管控 |
| **Agent** | 增强型搭档，执行层负责 |

## 2. 打磨阶段人机协作

### 2.1 打磨阶段角色分工

打磨阶段是"做正确的事"的核心环节，人类在目标设定和验收把关上深度参与，Agent 负责执行和信息聚合。

| 阶段 | 人类负责 | Agent 负责 |
|------|----------|------------|
| 需求收集 | ✅ 用户诉求原始语料收集 | - |
| 需求分析 | - | PO Agent：动机提取、线路图设计 |
| PRD 起草 | - | PO Agent：用户故事、验收标准 |
| 技术方案 | 验收 | PM Agent + Developer Agent：起草 |
| 用户方案 | 验收 | UI/UX Agent：起草 |
| 里程碑规划 | 验收 | PO Agent + PM Agent：起草 |

### 2.2 打磨阶段流程

```
打磨阶段完整流程
═══════════════════════════════════════════════════════════

需求收集  ←  人类负责（原始用户诉求）
     │
     ▼
需求分析  ←  PO Agent 负责
     │
     ▼
PRD 起草  ←  PO Agent 负责
     │
     ▼
╔═══════════════════════════════════════════════════════════╗
║  【Gate 1】PRD 验收 —— 逐步审查核对                       ║
║  关键内容：用户故事、线路图、验收标准                        ║
╚═══════════════════════════════════════════════════════════╝
     │ 通过
     ▼
技术方案起草  +  用户方案起草
     │
     ▼
╔═══════════════════════════════════════════════════════════╗
║  【Gate 2】方案验收 —— 逐步审查核对                       ║
║  关键内容：技术方案合理性、用户体验要求                    ║
╚═══════════════════════════════════════════════════════════╝
     │ 通过
     ▼
打磨阶段完成 → 进入执行阶段
```

## 3. 逐步审查核对模式

### 3.1 核心原则

**Agent 做"信息提取和呈现"，人类做"深度理解和判断"**，通过多轮对话逐步达成验收共识。

### 3.2 PRD 验收流程（Gate 1）

| 步骤 | Agent 行为 | 人类行为 |
|------|-----------|----------|
| 1 | 提取用户故事清单，展示核心场景 | 确认场景覆盖度 |
| 2 | 提取验收标准，标注可测试性 | 判断标准是否清晰、可验证 |
| 3 | 呈现用户线路图 | 确认路径是否完整、合理 |
| 4 | 记录人类反馈，生成修订建议 | 提出修改意见 |
| 5 | 迭代更新文档 | 复审确认通过 |

### 3.3 方案验收流程（Gate 2）

| 步骤 | Agent 行为 | 人类行为 |
|------|-----------|----------|
| 1 | 提取技术方案关键设计点 | 判断方案是否合理 |
| 2 | 提取用户方案核心设计决策 | 判断是否符合体验要求 |
| 3 | 呈现方案间的依赖关系 | 确认接口契约 |
| 4 | 记录人类反馈，生成修订建议 | 提出修改意见 |
| 5 | 迭代更新文档 | 复审确认通过 |

### 3.4 逐步审查核对示意

```
Agent 提取关键信息
        │
        ▼
人类围绕关键内容进行多轮沟通
        │
        ▼
逐步验收：
· 用户故事是否完整、清晰
· 线路图是否覆盖核心场景
· 验收标准是否可测试
· 技术方案是否合理
· 用户方案是否符合体验要求
        │
        ▼
人工确认通过 / 返回修改
```

## 4. 执行阶段人机协作

### 4.1 执行阶段特点

| 特点 | 说明 |
|------|------|
| **Agent 自主协作** | Developer Agent 与 Tester Agent 在 PM Agent 协调下自主执行 |
| **人类关键介入** | Gate 3（可使用软件验收）和 Gate 4（可维护软件验收）进行最终确认 |
| **PM Agent 协调** | 处理分歧、监控进度、触发升级 |

### 4.2 执行阶段流程

```
需求就绪
     │
     ▼
PM Agent 分配任务
     │
     ▼
┌─────────────────┐     ┌─────────────────┐
│ Developer Agent │◄───►│  Tester Agent   │
│   执行开发       │     │   执行测试       │
└────────┬────────┘     └────────┬────────┘
         │                      │
         │    PM Agent 协调     │
         │    （基于风险优先级） │
         │                      │
         └──────────┬───────────┘
                    │
                    ▼
            分歧解决机制
                    │
                    ▼
         ┌─────────────────┐
         │   上线发布       │
         └────────┬────────┘
                  │
                  ▼
         【Gate 3】可使用软件验收
         单元测试 + E2E测试 + 人工验收
                  │ 通过
                  ▼
         ┌─────────────────┐
         │   文档完善       │
         └────────┬────────┘
                  │
                  ▼
         【Gate 4】可维护软件验收
         已部署软件 + 使用说明文档
                  │ 人类最终确认
                  ▼
              项目完成
```

## 5. 分歧处理机制

### 5.1 分歧类型

| 类型 | 说明 |
|------|------|
| **Developer-Tester 分歧** | 功能实现与测试标准的不一致 |
| **Agent-Agent 分歧** | 不同 Agent 之间的理解差异 |
| **方案分歧** | 技术方案或用户方案的不一致 |

### 5.2 分级处理

| 风险等级 | 处理方式 | 决策方 |
|----------|----------|--------|
| **低风险** | Agent 自行协调 | Agent |
| **中风险** | PM Agent 介入协调 | PM Agent |
| **高风险** | 升级人类 PM 决策 | 人工 PM |

### 5.3 分歧处理流程

```
分歧发生
    │
    ▼
评估风险等级
    │
    ├── 低风险 → Agent 自行协调
    │              │
    │              ▼
    │         更新状态
    │
    ├── 中风险 → PM Agent 介入协调
    │              │
    │              ▼
    │         提出解决方案
    │              │
    │              ▼
    │         人类确认（如需要）
    │
    └── 高风险 → 升级人工 PM
                   │
                   ▼
              人类决策
                   │
                   ▼
              执行并记录
```

### 5.4 Developer-Tester 特定处理

当 Developer Agent 与 Tester Agent 出现分歧时：
- 由 PM Agent 根据风险优先级处理
- 优先基于已批准的文档
- 其次基于最后更新
- 争议升级人工 PM

## 6. 升级机制

### 6.1 升级触发条件

| 条件 | 阈值 | 说明 |
|------|------|------|
| 阻塞项时长 | > 4 小时（警告）/ > 24 小时（升级） | 阻塞任务继续执行的问题 |
| 任务延期 | > 1 天（警告）/ > 3 天（升级） | 计划延迟 |
| 测试失败率 | > 10%（警告）/ > 30%（升级） | 质量风险 |
| 风险概率变化 | +10%（警告）/ +30%（升级） | 风险升级 |

### 6.2 升级处理流程

```
检测到升级条件
    │
    ▼
PM Agent 记录并通知
    │
    ▼
评估是否可自行解决
    │
    ├── 可解决 → 协调解决，更新状态
    │
    └── 不可解决 → 升级人工 PM
                      │
                      ▼
                 人工决策
                      │
                      ▼
                 执行并记录
```

## 7. 重做机制

### 7.1 重做触发

打磨阶段任一 Gate 验收不通过时触发重做机制。

### 7.2 重做流程

```
Gate 验收不通过
    │
    ▼
记录不通过原因
    │
    ▼
Agent 承担重做成本
    │
    ▼
重新走对应流程
    │
    ▼
人类在新一轮迭代中逐步验收
```

### 7.3 重做成本分配

| 阶段 | 重做成本承担 |
|------|-------------|
| 打磨阶段 | Agent 承担 |
| 执行阶段 | 按责任划分 |

## 8. 协作协议

### 8.1 基本协议

| 协议 | 说明 |
|------|------|
| **消息格式** | 统一使用 JSON，字段见各接口定义 |
| **响应时效** | 紧急（阻塞）< 5min，建议 < 1h，状态更新 < 24h |
| **确认机制** | 重要消息需接收方确认（`ack: true`） |
| **上下文传递** | 跨 Agent 共享项目上下文，通过知识库而非消息链 |
| **冲突解决** | 优先基于已批准的文档，其次基于最后更新，争议升级人工 PM |

### 8.2 UI/UX 协作补充协议

| 协议 | 说明 |
|------|------|
| **设计交付物标准** | UI/UX Agent 提交的设计稿需包含标注规范、尺寸要求、交互说明 |
| **设计评审规则** | 设计初稿完成后必须经过相关干系人评审，评审通过方可进入开发 |
| **设计变更管理** | 设计定稿后的变更需评估影响范围，重大变更需重新评审 |
| **与 Developer 协作** | 设计稿需同步给 Developer Agent，提供设计标注和交接说明 |
