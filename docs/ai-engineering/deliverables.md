## 1. 概述

本文档定义 AI 软件研发流程中各阶段的关键文档（产出物），包括：
- 文档内容要求
- 验收标准
- 责任人
- 存放位置

## 2. 打磨阶段关键文档

### 2.1 PRD（产品需求文档）

| 属性 | 说明 |
|------|------|
| **责任人** | PO Agent（起草）、Product Owner（验收） |
| **参与 Gate** | Gate 1 |
| **存放位置** | `docs/product/PRD.md` |

#### 内容要求

| 内容 | 说明 | 验收标准 |
|------|------|----------|
| **用户故事** | 以用户视角描述功能需求 | 完整、清晰、可测试 |
| **用户线路图** | 用户使用产品的路径图 | 覆盖核心场景 |
| **验收标准** | 判断需求是否完成的准则 | 可测试、无歧义 |
| **原型图** | 产品界面原型设计 | 覆盖核心页面、符合需求 |

#### 用户故事模板

```markdown
## 用户故事：[功能名称]

### 背景
[为什么需要这个功能]

### 用户需求
[用户想要什么]

### 验收标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

### 优先级
[Must / Should / Could / Won't]
```

### 2.2 技术方案

| 属性 | 说明 |
|------|------|
| **责任人** | Orchestrator Agent + Full-stack Developer Agent（起草）、Product Manager（验收） |
| **参与 Gate** | Gate 2 |
| **存放位置** | `docs/engineering/` |

#### 内容要求

| 内容 | 说明 | 验收标准 |
|------|------|----------|
| **系统设计** | 架构设计、模块划分 | 清晰、可实现 |
| **技术选型** | 技术栈选择、依赖说明 | 合理、有依据 |
| **接口契约** | 模块间接口定义 | 完整、无歧义 |

### 2.3 用户方案

| 属性 | 说明 |
|------|------|
| **责任人** | UI/UX Agent（起草）、Product Owner（验收） |
| **参与 Gate** | Gate 2 |
| **存放位置** | `docs/design/` |

#### 内容要求

| 内容 | 说明 | 验收标准 |
|------|------|----------|
| **设计规范** | 组件库、颜色、字体、间距 | 完整、一致 |
| **设计稿** | UI 设计稿 | 覆盖所有状态 |
| **交互说明** | 组件状态、过渡动画、微交互 | 清晰、可实现 |

#### 设计交付物标准

| 交付物 | 说明 | 格式要求 |
|--------|------|----------|
| **设计稿** | 完整的 UI 设计稿 | Figma/Sketch/Webflow 链接 |
| **标注规范** | 间距、颜色、字体等规范说明 | 设计稿内嵌或独立文档 |
| **尺寸规格** | 响应式断点、适配说明 | 包含桌面/平板/手机 |
| **交互说明** | 组件状态、过渡动画、微交互 | 独立文档或设计稿注释 |
| **图标资源** | 图标源文件或图标库引用 | SVG/PNG + 名称映射 |
| **设计标注** | 开发必需的标注信息 | Zeplin/Figma 导出或独立文档 |

### 2.4 里程碑规划

| 属性 | 说明 |
|------|------|
| **责任人** | PO Agent + Orchestrator Agent（起草）、Product Owner（验收） |
| **参与 Gate** | Gate 2 |
| **存放位置** | `docs/project-management/milestones.md` |

#### 内容要求

| 内容 | 说明 | 验收标准 |
|------|------|----------|
| **阶段划分** | 各阶段的起止时间和目标 | 合理、可执行 |
| **交付物清单** | 各阶段需要产出的文档/代码 | 完整、无遗漏 |
| **验收标准** | 各阶段完成的判定条件 | 清晰、可测试 |

## 3. 执行阶段关键文档

### 3.1 任务清单

| 属性 | 说明 |
|------|------|
| **责任人** | Orchestrator Agent（维护） |
| **存放位置** | `docs/project-tasks/W{nn}/tasklist.md` |

#### 内容要求

| 内容 | 说明 |
|------|------|
| **任务列表** | 当前周需要完成的任务 |
| **任务状态** | 待开始 / 进行中 / 待验收 / 完成 |
| **负责人** | 认领任务的 Agent 或人类 |
| **截止时间** | 计划完成时间 |

### 3.2 Issue 管理

| 属性 | 说明 |
|------|------|
| **责任人** | Orchestrator Agent（维护）、Full-stack Developer Agent / Tester Agent（执行） |
| **创建时机** | Gate 2 验收通过后，Phase 执行启动前 |
| **存放位置** | GitHub Issues（纳入对应的 Milestone） |

#### Issue 统一模板

每个 Issue 必须包含以下字段：

```markdown
## 目标
[一句话描述本 Issue 要达成的目标]

## 背景
[为什么需要做这件事？关联的用户故事或里程碑]

## 执行计划
- [ ] 步骤 1
- [ ] 步骤 2
- [ ] 步骤 3

## 验收标准
### 功能验收项
- [ ] 标准 1：可测试、可验证
- [ ] 标准 2
- [ ] 标准 3
### 测试要求
- [ ] 单元测试覆盖度 ≥ 80%
- [ ] 边界条件已覆盖：1) [场景] 2) [场景]

## 产出物
[本 Issue 完成后的交付物（代码、文档、测试报告等）]

## 依赖
- dependsOn: [前置 Issue 编号列表，如无则填「无」]
- blockedBy: [外部阻塞条件，如无则填「无」]
- criticalChain: true / false
```

#### 测试责任划分

| 测试类型 | 归属 | 责任人 |
|----------|------|--------|
| **单元测试** | 功能 Issue 内 | Developer |
| **集成测试** | 独立 Issue（`{Phase}-T`） | Developer / Tester |
| **E2E 测试** | 独立 Issue（`{Phase}-T`） | Developer + 人类确认场景 |

#### 管理规则

| 规则 | 说明 |
|------|------|
| **关联 Milestone** | 每个 Issue 必须关联到对应的里程碑 |
| **验收标准可测试** | 验收标准必须可验证、可测试，不能是模糊描述 |
| **优先级标注** | 使用 P0/P1/P2 标签标注优先级 |
| **类型分类** | 使用 Bug/Feature/Task 标签标注类型 |
| **Sub-issue 拆分** | 超过 3 天的工作量必须拆分为 Sub-issue |

### 3.3 看板管理

| 属性 | 说明 |
|------|------|
| **责任人** | Orchestrator Agent（维护） |
| **创建时机** | Gate 2 验收通过后 |
| **存放位置** | GitHub Projects（Kanban 视图） |

#### 看板状态列

| 状态 | 说明 | 准入条件 |
|------|------|----------|
| **待办（To Do）** | 已创建但未开始的 Issue | Issue 已关联 Milestone，验收标准完整 |
| **进行中（In Progress）** | 正在处理的 Issue | 已分配负责人，执行计划明确 |
| **待验收（In Review）** | 已提交等待验收的 Issue | 代码已提交，自测通过 |
| **已完成（Done）** | 验收通过的 Issue | 验收标准全部满足 |

#### 管理规则

| 规则 | 说明 |
|------|------|
| **WIP 限制** | 每个 Agent 同时「进行中」的 Issue 不超过 3 个 |
| **状态同步** | Issue 状态变更时，看板状态列同步更新 |
| **更新频率** | 每日至少更新一次看板状态 |
| **Phase 切换** | 当前 Phase 结束后，看板归档，下一 Phase 新建看板 |
| **阻塞标记** | 被阻塞的 Issue 添加 `blocked` 标签并在看板中标记 |

### 3.4 问题记录

| 属性 | 说明 |
|------|------|
| **责任人** | Orchestrator Agent（维护） |
| **存放位置** | `docs/project-tasks/W{nn}/problems/` |

#### 内容要求

| 内容 | 说明 |
|------|------|
| **问题描述** | 问题的详细描述 |
| **严重程度** | P0 / P1 / P2 / P3 |
| **状态** | 待处理 / 处理中 / 已解决 |
| **解决方案** | 解决措施（若有） |

## 5. 状态跟踪文档

### 5.1 STATUS.md

| 属性 | 说明 |
|------|------|
| **责任人** | Orchestrator Agent（自动更新）、人工 PM（审批重大变更） |
| **存放位置** | `docs/STATUS.md` |

#### 内容要求

| 内容 | 说明 | 更新频率 |
|------|------|----------|
| **项目状态** | 当前阶段、进行中事项 | 按需 |
| **里程碑进度** | 各阶段完成情况 | 按周 |
| **阻塞项** | 当前阻碍进度的问题 | 按需 |
| **最近更新** | 任务完成时的变更记录（自动追加） | 任务完成时自动触发 |
| **自动化设置** | 自动提交等特性的启用状态和累计确认次数 | 按需 |

**最近更新自动追加格式**：

```markdown
| {时间} | #{Issue编号} | {type}: {描述} | {Agent角色} |
```

**自动化设置模板**：

```markdown
## 自动化设置

| 特性 | 状态 | 累计确认次数 |
|------|------|-------------|
| 任务完成自动提交 | disabled / enabled | {次数} |
```

### 5.2 周报

| 属性 | 说明 |
|------|------|
| **责任人** | Orchestrator Agent（起草） |
| **存放位置** | `docs/project-tasks/W{nn}/STATUS.md` |

#### 内容要求

| 内容 | 说明 |
|------|------|
| **本周进展** | 完成的任务、达成的里程碑 |
| **下周计划** | 计划开始的任务 |
| **风险状态** | 当前风险及应对措施 |
| **阻塞项** | 需要协调解决的问题 |

## 6. 文档验收要点

### 6.1 PRD 验收清单

- [ ] 用户故事覆盖所有核心功能
- [ ] 用户线路图包含主要使用路径
- [ ] 验收标准可测试、无歧义
- [ ] 优先级划分合理
- [ ] Non-Goals 与验收标准无冲突
- [ ] 关键术语在全文定义一致

### 6.2 技术方案验收清单

- [ ] 系统架构清晰合理
- [ ] 技术选型有充分依据
- [ ] 接口定义完整、无歧义
- [ ] 考虑了扩展性和维护性

### 6.3 用户方案验收清单

- [ ] 设计稿覆盖所有页面和状态
- [ ] 标注规范完整
- [ ] 交互说明清晰
- [ ] 符合设计系统规范
